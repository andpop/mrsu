## Лабораторная работа 3. Разработка тест-кейсов на основе чек-листов
### Цель работы
Научиться детализировать проверки из чек-листов в полноценные, структурированные тест-кейсы, содержащие чёткие шаги и ожидаемые результаты. Закрепить навыки проектирования тестовой документации, соответствующей различным типам тестирования (функциональное, нефункциональное, дымовое), и организованной в логические наборы.


### Задание
На основе чек-листов, разработанных вами во второй лабораторной работе для игры «Сапёр», составьте наборы тест-кейсов.

Каждый чек-лист должен быть преобразован в отдельный набор тест-кейсов, соответствующий его типу:

1. Набор тест-кейсов: Позитивные сценарии функционального тестирования
2. Набор тест-кейсов: Негативные сценарии функционального тестирования
3. Набор тест-кейсов: Нефункциональное тестирование
4. Набор тест-кейсов: Дымовое тестирование (Smoke Test)

### Требования к оформлению тест-кейсов
Каждый тест-кейс должен содержать следующие поля:

* Название — краткое и понятное описание цели проверки (например: «Проверка победы при открытии всех безопасных клеток»).
* Тип тестирования — указывается явно: Функциональное (позитивное), Функциональное (негативное), Нефункциональное или Дымовое.
* Предусловия (если необходимы) — например: «Игра запущена», «Выбран уровень “Новичок”».
* Шаги — пошаговое описание действий тестировщика (нумерованный список).
* Ожидаемый результат — чёткое и проверяемое описание того, что должно произойти после выполнения шагов.
Примечание: Все тест-кейсы должны быть основаны на спецификации из ЛР №1 и чек-листах из ЛР №2. Не допускается дублирование одинаковых тест-кейсов в разных наборах без обоснования.

### Требования к оформлению отчёта
* Все тест-кейсы оформляются в одном файле test-cases.docx.
* Файл должен содержать четыре раздела, каждый из которых соответствует одному из наборов (см. выше).
* В начале каждого раздела укажите краткое описание его назначения (например: «Набор содержит тест-кейсы для проверки корректной работы игры при штатном использовании»).
* Используйте таблицу для оформления каждого тест-кейса.
 
### Пример оформления тест-кейса
| Поле                    | Значение                                                                                                                                        |
|-------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------|
| **Название**            | Проверка сохранения статистики после перезагрузки страницы                                                                                      |
| **Тип тестирования**    | Функциональное (позитивное)                                                                                                                     |
| **Предусловия**         | Игра запущена, сыграна хотя бы одна партия                                                                                                      |
| **Шаги**                | 1. Завершите одну игру (любой исход — победа или поражение).<br>2. Обновите страницу браузера.<br>3. Перейдите в раздел отображения статистики. |
| **Ожидаемый результат** | Статистика отображает корректное количество сыгранных игр, побед и поражений, соответствующее состоянию до перезагрузки.                        |

Примечание: При оформлении нескольких тест-кейсов подряд используйте отдельную таблицу для каждого кейса, чтобы обеспечить читаемость и возможность независимого выполнения.

### Рекомендации
* Убедитесь, что в наборах учтены все уровни сложности («Новичок», «Любитель», «Профессионал»), если это применимо.
* Для нефункциональных тест-кейсов укажите измеримые ожидания (например: «Время отклика интерфейса после клика — не более 100 мс»).
* В дымовом наборе ограничьтесь 5–8 критически важными тест-кейсами, подтверждающими работоспособность основного сценария игры.
* Проверьте, что негативные тест-кейсы действительно проверяют устойчивость системы к ошибкам пользователя, а не просто дублируют позитивные.
